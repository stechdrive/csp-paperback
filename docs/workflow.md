# アニメ制作ワークフローとこのツールの位置づけ

## 前提：なぜ静止画が必要か

クリスタはアニメーション制作ツールだが、工程の多くは静止画単位で動く。

- 動画・仕上げ工程はクリスタを使わないことが多い
- 撮影工程はそもそも静止画の合成
- 次工程が紙（印刷・転写）の場合もある
- 外部スタッフがクリスタを持っていない場合もある

つまり **「クリスタで作ったアニメーション素材を、静止画として正しく取り出す」** というニーズは制作フローの構造的な問題として存在する。

---

## 現場の現実：別トラック方式

クリスタのアニメーション書き出しは「セル単体」を出力する設計になっている。
そのため、原画・演出修正・作監修正など複数の工程が絡む場合、現場では以下のような構造をとる。

```
原画トラック（A）
  1番  2番  3番  ...

演出修正トラック（A_en）
  1番（修正あり）  3番（修正あり）  ...

作監修正トラック（A_animation_check）
  2番（修正あり）  ...
```

修正が発生したキーに、修正工程のトラックで同じ番号のセルを置く。
クリスタの書き出しで自動化でき、静止画として次工程に渡せる。

ただし「このキーの作監修正だけ確認したい」「原画と修正を重ねた完成画像を見たい」
といった用途には対応できない。各トラックの書き出しが独立しているから。

---

## 理論上の理想：セル内包型

各工程をセルの中に従属させる構造。

```
原画トラック（A）
  1番/
    原画レイヤー
    _演出修正/
      ...
    _作監修正/
      ...
```

構造として直感的だが、クリスタの書き出しでは工程ごとの独立した出力ができない。
全工程を合成した画像が1枚出るだけで、修正指示を単独で確認することができない。
また、作業途中に原画の不透明度を下げた状態で保存していると、その影響が出力に混入する。

---

## このツールが解決すること

両方のワークフローに対応し、どちらで作られたデータからも工程ごとの完成画像を正しく取り出せる。

**別トラック方式への対応**
- XDTSまたはCSPBからタイムライン情報を読み込み、複数トラックのセルを同期して合成
- 「原画トラックのセル＋BG＋コンテキストレイヤー」を1枚の画像として出力

**セル内包型への対応**
- セル内の工程フォルダ（processTableにマッチするフォルダ名）を検出して独立出力
- 工程フォルダ本体は100%固定、内部の作業レイヤーは不透明度を尊重
- 各工程を単独で確認できる完成画像として出力

**共通の前提**
- アニメーションフォルダ・セルフォルダは「構造」として100%固定扱い
- 作業者が参考素材を半透明にして作業した状態でも、出力は「紙に刷り上がった状態」になる
